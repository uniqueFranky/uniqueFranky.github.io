符号执行 Symbolic Execution

定义

每次分析程序中的一条路径，按照某种顺序遍历程序中的路径。在分析过程中，给所有输入变量赋予一个符号值而非具体值，借助 SMT Solver 来实现路径可达性、路径执行前提、执行结果的分析。

无穷多路径

当程序中存在循环、递归时，可能会出现有无穷多的执行路径，这种情况一般遍历有限次来进行模拟。

调用约束求解的时机

求解策略

调用 SMT Solver 进行求解的时机分为两种：

  • Eager Evaluation 每次进行分支时就调用 SMT Solver 对路径的 Predicate进行求解，如果不可满足，则该路径是 infeasible 的，无需继续执行

  • Lazy Evaluation 在每条路径的末尾对完整路径的 Predicate 进行求解

其中 Eager Evaluation 会调用更多次数的 SMT Solver ，但是产生的路径数量较少；Lazy Evaluation 调用 SMT Solver 的次数更少，且如果路径不可达，则路径的 Predicate 更有可能有较多的冲突， SMT Solver 求解速度可能更快，但是需要探索的路径更多。

动态符号执行 Dynamic Symbolic Execution

Motivation

当出现以下情况时， SMT Solver 可能无法完成求解：

  • 复杂 predicate：

    • p->next->value == x

  • 发生系统调用

    • file.read() == x

为了解决这些无法求解的 predicate ，可以将程序动态运行时的实际值代入，即用真实值代替符号值，从而使得 SMT Solver 能够求解这些约束。

流程

在程序开始执行之前，除了给输入变量赋予符号值之外，同时给他们赋予随机的实际值，然后使用这些实际值进行程序的执行，同时对符号值记录其进行的运算，累计路径的 predicate 。

与静态符号执行不同的是，静态符号执行在进行分支时，可以对所有的分支进行探索，也就是由符号执行决定探索的分支；而动态符号执行则以实际值来进行分支的选取，只能选择一个分支，也就是动态执行决定探索的分支。

但是这样一来，岂不是只能执行众多路径中的一条了？为了能够遍历程序中所有的路径，我们开始尝试寻找新的实际值，使得使用这些实际值从头开始执行时能够执行到程序中不同的路径。

具体来说，对于我们累计的路径 predicate ，一定是一些原子 predicate 的合取式，我们从这个合取式中取一个还没有被取反过的原子 predicate ，将其取反后得到新的合取式，并将新的合取式扔给 SMT Solver ，将会得到一组新的实际值，使用这组实际值从头开始执行程序将保证执行到另一条路径。

代入实际值

当遇到 SMT Solver 不支持的表达式时，我们将 SMT Solver 不认识的部分替换为动态执行时的实际值，从而让 SMT Solver 来解决这一表达式。比如我们有这样的 predicate ： $(bb)\%50==ab5(55)\%50==a$ ，即可被 SMT Solver 求解。

静态符号执行 vs 动态符号执行

在动态符号执行时，遇到 SMT Solver 不可解的问题，将符号值替换为实际值，有可能会使原本可满足的约束变得不可满足，如： ，若将 的实际值 代入，该约束将变得不可满足，但是原约束是可以满足的；但是遭遇这种问题时，一定是遇到了 SMT Solver 不可解问题，静态符号执行将对这种问题束手无策。因此动态符号执行的效果一定优于静态符号执行。

当遇到约束无法求解的情况时，为什么不是将符号值替换为随机值，而是替换为具体值？这就和动态符号执行的 motivation 有关了，与系统环境有关的值，比如系统调用的返回值，被替换为随机值将会使得程序分析没有意义。
